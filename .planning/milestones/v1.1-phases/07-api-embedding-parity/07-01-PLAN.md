<objective>
Add semantic embedding generation to REST API create and update handlers using TDD, closing the parity gap between MCP tools and the REST API.

Purpose: Memories created or updated via the REST API are currently invisible to semantic search because `api_add` and `api_update` in `api.py` do not call `_embed_and_store` after writing to SQLite — while the MCP tool equivalents (`memory_add` and `memory_update` in `tools.py`) do. This plan adds the missing calls and proves correctness via integration tests.

Output: 5 passing integration tests + 2 production code changes (3 lines total) in api.py
</objective>

  <behavior>
    EMBD-01 — POST /api/memories creates a memories_vec row:
    - POST /api/memories with {"content": "test"} → 201 + row in memories_vec with matching rowid
    - The embedding rowid matches the memory's SQLite rowid (foreign key integrity)

    EMBD-02 — PUT /api/memories/{id} re-embeds on content change:
    - PUT /api/memories/{id} with {"content": "new content"} → memories_vec row updated
    - PUT /api/memories/{id} with {"tags": ["new"]} (no content) → memories_vec row unchanged

    Parity — REST and MCP memories are equally findable:
    - Create one memory via REST API, one via MCP tool → _semantic_search("query") returns both
  </behavior>
  <implementation>
    RED phase:

    1. Add `db_conn_with_vec` fixture to `tests/conftest.py`:
       - Copy the `db_conn` fixture pattern but add sqlite-vec loading before `_ensure_schema`:
         ```python
         import sqlite_vec
         db.enable_load_extension(True)
         sqlite_vec.load(db)
         db.enable_load_extension(False)
         ```
       - Monkeypatch all four modules' `_get_db` (same as `db_conn`)
       - Place it directly after the `db_conn` fixture in the file

    2. Add 5 test functions to `tests/test_api.py`:

       a. `test_api_add_creates_embedding(db_conn_with_vec, mock_embedder, monkeypatch)`:
          - Patch IMPORT_ROOTS to include /tmp
          - Build TestClient from `_build_api_app()`
          - POST /api/memories with content
          - Assert 201 status
          - Query `SELECT rowid FROM memories WHERE id = ?`
          - Query `SELECT rowid FROM memories_vec WHERE rowid = ?`
          - Assert vec_row is not None

       b. `test_api_add_embedding_rowid_matches_memory(db_conn_with_vec, mock_embedder, monkeypatch)`:
          - Same setup as (a)
          - POST /api/memories
          - Get memory rowid and vec rowid
          - Assert they are equal

       c. `test_api_update_content_regenerates_embedding(db_conn_with_vec, mock_embedder, monkeypatch)`:
          - POST a memory, record its vec rowid
          - PUT /api/memories/{id} with new content
          - Query memories_vec — row should still exist (rowid unchanged since it's an upsert by rowid)
          - Assert the vec row is present

       d. `test_api_update_no_content_preserves_embedding(db_conn_with_vec, mock_embedder, monkeypatch)`:
          - POST a memory, record vec row presence
          - PUT /api/memories/{id} with only tags (no content field)
          - Assert vec row still exists and was not removed/changed
          - Use mock_embedder call tracking or simply verify row still present

       e. `test_rest_and_mcp_memories_equally_findable_by_semantic_search(db_conn_with_vec, mock_embedder, monkeypatch)`:
          - POST /api/memories to create a REST memory
          - Directly call `_embed_and_store` for an MCP-style memory (insert row + embed)
          - Call `_semantic_search(db, "query", limit=10)` from `remind_me_mcp.db`
          - Assert both memory IDs appear in results
